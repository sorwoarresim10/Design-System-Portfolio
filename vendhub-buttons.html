01 · Audit & Problems

Audit & Problems

Issues identified by comparing the existing button system against VendHub's live components and design tokens.

1.1 Typography ConflictsCritical
Critical
Sub-scale font sizes in button specs
The button spec defines an xs size at 10px and an sm size at 12px. However, the current typography system renders 10px text via text-[10px] — an arbitrary value flagged in the audit as an anti-pattern (25 occurrences). The smallest named scale step is text-xs = 12px.
spec says 10px→min 12px (text-xs)
High
Inconsistent font weight on buttons
Button labels across the app use font-medium (49% of all weight usages) and font-semibold (27%) with no documented rule for buttons. Some instances use font-bold. The spec should standardise on font-semibold / 600 for all button labels.
mixed: 500/600/700→always 600
1.2 Color System ConflictsCritical
⚠️
Primary color identity split: The shadcn --primary HSL token resolves to teal while bg-primary resolves via --primary500 to green. Buttons may render in either colour depending on whether they use a shadcn variant or custom classes. All buttons must use --primary-600 (#16A34A) as the single source of truth.
Critical
Warning buttons use un-tokenised orange
Warning variant buttons (Link to Customer, View Details, Pay via Credit Card) show an orange color in the UI. However --warning, --warning-foreground tokens are not defined in :root — they produce invalid CSS. Buttons currently hardcode bg-orange-500 bypassing the token system entirely.
bg-orange-500→bg-[--warning-600]
Critical
Hardcoded hex values in button backgrounds
~385 hardcoded hex instances exist in the codebase. Several button-adjacent elements use raw hex (e.g. #15A34A, #DC2626) instead of semantic tokens. A brand colour change would require hunting down each occurrence.
#15A34A hardcoded→var(primary)
High
Disabled state uses raw gray rather than design token
Disabled buttons show approximately bg-gray-200 / text-gray-500 but this is not standardised. Some disabled buttons render with a green-tinted background because the base primary style applies opacity instead of overriding the background.
High
No semantic success button variant
The design system defines a full success scale (50–600) and the UI shows success states in banners, but there is no success button variant documented or implemented. Confirm/approve actions fall back to using the primary green style.
1.3 Missing & Inconsistent StatesCritical
Critical
Zero active/pressed states across all variants
The audit confirms: "No active/pressed states anywhere — zero components define active: pseudo-class styles." When a user clicks a button there is no visual depression or scale feedback. Every button variant must define active:scale-[0.975] and active:brightness-95.
Critical
Four different focus ring patterns
Button focus rings vary across the system: ring-1 primary-500, ring-2 ring offset-2, focus vs focus-visible. Badge uses focus: (fires on mouse click). Dialog close button has no focus ring at all.
High
No loading state defined
The existing button spec documents size, variants, and some states, but there is no documented loading state. Forms, async actions need a spinner-in-button pattern with disabled pointer events to prevent double submission.
1.4 Icon Usage & NamingHigh
High
Two icon libraries with no documented convention
36 files import from both lucide-react and @fluentui/react-icons. The same icon concept (Search, X, Check) is sourced from different libraries in different buttons causing visual inconsistency.
lucide + fluent mixed→lucide-react as default
High
Stepper & Add-to-Cart are undocumented special types
The stepper (quantity control) and the Add to Cart composite button appear throughout the product but have no formal specification. They have their own layout rules, circular minus/plus buttons, and disabled states that differ from standard buttons.

2.8 Accessibility Requirements
✓
Colour contrast 4.5:1 minimum — All text in buttons must pass WCAG AA. Primary (#16A34A on white) passes. Verify disabled state (#9CA3AF text on #F3F4F6) — if it fails, add a tooltip explaining why the button is disabled.
✓
Use focus-visible, not focus — focus: fires on mouse click causing ring flash. Every button must use focus-visible:ring-2 focus-visible:ring-primary-600 focus-visible:ring-offset-2 and focus-visible:outline-none.
✓
Icon-only buttons require aria-label — Every icon button without visible text must have a descriptive aria-label. Do not rely on title attribute — it is not announced by all screen readers.
✓
Loading state: disable + announce — When a button triggers an async action, set disabled (or aria-disabled="true") and add aria-busy="true". Update the label to reflect state (e.g. "Saving…").
✓
Minimum 44×44px touch target — Buttons smaller than 44px visually must be wrapped in a container that expands the clickable area, or have sufficient padding. Use xs buttons only in desktop-only admin interfaces.
i
Never use <div> or <a> styled as buttons for actions — Use <button type="button"> for in-page actions and <a href="…"> only for genuine navigation. <a> elements do not have the same keyboard behaviour as <button>.
i
Disabled context — Avoid relying on colour alone to communicate a disabled state. Add a tooltip or adjacent note explaining why the action is unavailable (e.g. "Complete step 2 to continue").
03 · Migration, Governance & Usage

Migration, Governance & Usage

How to transition to the updated system, what to deprecate, and how to keep the system healthy going forward.

3.1 Migration Phases
1
Fix the token foundation — Day 1, ~1 hour
Add missing CSS variables and fix the broken ones before touching any component. This unblocks all semantic Tailwind classes like bg-warning, bg-error, bg-info. Note: --primary maps to the hover state; the button default uses primary-600 = #16A34A directly via the hex token, not via this HSL variable.
/* globals.css — add to :root after existing shadcn tokens */ --error: 0 72% 51%; --error-foreground: 0 0% 100%; --warning: 32 85% 53%; --warning-foreground: 0 0% 100%; --info: 199 89% 48%; --info-foreground: 0 0% 100%; /* Fix success-foreground + unify primary identity to green */ --success-foreground: 0 0% 100%; --primary: 142 72% 29%;
2
Update button.tsx base component — Day 1, ~30 min
Add the missing states to the shadcn Button component. This single file change propagates to every button across the app.
// components/ui/button.tsx — add to buttonVariants base class const baseClasses = ` active:scale-[0.975] active:brightness-95 focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-primary-600 focus-visible:ring-offset-2 transition-all duration-100 ` // Add disabled overrides to ALL variants (replace opacity approach): disabled: "bg-neutral-100 text-neutral-400 border border-neutral-200 shadow-none cursor-not-allowed pointer-events-none"
3
Fix tooltip and hardcoded component colors — Day 1, ~20 min
These are fast targeted fixes in 5 specific files that have the highest visual impact.
// tooltip.tsx — green tooltips → neutral dark bg-primary text-primary-foreground → bg-neutral-900 text-white // select.tsx — hardcoded white → token bg-white → bg-popover // dropdown-menu.tsx — hardcoded white → token bg-white → bg-popover // select.tsx check icon — bypass → token text-green-500 → text-primary
4
Migrate remaining patterns — Ongoing, assigned per PR
Search for bg-orange-* and text-orange-* in button contexts and replace with the now-defined warning tokens (~92 orange usages in the audit). Update Badge, Select, Tabs, Switch, Slider, and Dialog close button to the unified focus ring pattern. Work file by file through the ~385 hardcoded hex instances, prioritizing button-adjacent components. Also replace any instances of bg-blue-* used for Contact buttons with the info token bg-[#2563EB] or the new info variant class once defined.

3.3 Variant Hierarchy & Usage Guidelines
✓ Do
One primary button per view or dialog. It should represent the single most important action.
Use neutral for cancel, go back, or any secondary action that appears alongside a primary button.
Use primary-outline when you need a second emphasized action but don't want two solid green buttons competing.
Reserve destructive solid for the confirmation step inside a dialog — not for the trigger that opens the dialog.
Use ghost for tertiary navigation actions (View All →, Change) that don't need to look like buttons at rest.
Default button size is lg (40px). Use md inside tables, toolbars, and card footers.
✗ Don't
Never show two primary buttons side by side — one will always be more important than the other.
Never use warning as a replacement for primary — it implies risk. It is not interchangeable with orange-branded actions.
Never hide the reason a button is disabled. Always add context adjacent to the button or in a tooltip.
Never use xs buttons for destructive or high-stakes actions — the touch target is too small and the text too small to read in a stressed state.
Never style <a> links to look like primary buttons unless they genuinely navigate to a new page. Use <button> for all interactive actions.

3.4 Governance Rules
✓
Never add a new button color without a token. If a new context requires a button color not covered by the existing variants, create the CSS variable first in globals.css, then expose it in tailwind.config.ts, then create the variant. No hardcoded hex values in component files.
✓
New button variants must include all 6 states. Default, hover, active, focus, loading, and disabled. A variant is not complete until all states are defined and reviewed. Include visual examples in PR description.
✓
Use the design system's icon library. Default to lucide-react for all button icons. Only use @fluentui/react-icons for icons that have no lucide equivalent and have been approved. Document the exception in a comment.
✓
Button size defaults to lg. When adding a new button, start with size="lg". Only deviate if the context clearly requires a different size (toolbar = md, hero CTA = xl). Documenting the size decision prevents drift over time.
i
PR checklist for button changes: (1) Does it use design tokens only? (2) Does it have all 6 states? (3) Does it pass 4.5:1 contrast? (4) Is focus-visible (not focus) used? (5) Are icon-only buttons labelled? (6) Are disabled states explained in context?
i
Special button types (stepper, dropdown, add-to-cart) live in components/ui/. They are not ad hoc implementations scattered across feature components. Any changes to special button behaviour must be made to the shared component, not to a one-off copy.
✗
Never create a "custom" button by composing raw divs. Every clickable element that performs an action must be a <button> or use role="button" with keyboard handling. This is both an accessibility requirement and a design system hygiene rule.
