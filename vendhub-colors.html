01 · Problems

10 Issues Found in the Current System

Audit of globals.css (562 lines, 222 CSS variables) and tailwind.config.ts revealed critical token bugs, identity conflicts, and structural problems.

🔴 P0 — Primary Identity Conflict
--primary (shadcn HSL, line 76) resolves to teal (#2a8a7e). --primary500 (line 106) is green (#16A34A). shadcn components get teal; custom code using bg-primary gets green. Two different brand colors in production.
🔴 P0 — 6 Broken Tokens
--error, --error-foreground, --warning, --warning-foreground, --info, --info-foreground are referenced in tailwind.config.ts but never defined in :root. Every bg-error, bg-warning, bg-info class produces invalid CSS.
🔴 P0 — Success Token Bugs
--success: 143, 35%, 44% uses commas in HSL (invalid modern CSS — must be spaces). Worse: --success-foreground is set to the same value as --success. White-on-success text renders as green-on-green.
🟠 P1 — 79% Raw Tailwind Colors
Only 21% of color usage goes through the custom token system. text-gray-* (3,602), bg-gray-* (456+) bypass every token. The system exists but is largely ignored.
🟠 P1 — Dual Text Color Systems
text-gray-* (Tailwind defaults): 2,193 uses. text-typography-* (custom tokens): 384 uses. Two systems doing the same job, no consistency, impossible to theme.
🟠 P1 — Orphaned Tertiary Scale
13 --tertiary-* tokens defined (orange scale, #fffaf5 → #543112). Zero usage across entire codebase. Nearly identical to the warning scale (tertiary500 #e78128 vs warning500 #e77828 — 1 hex digit difference). Fully orphaned dead code.
🟠 P1 — Broken Secondary Neutral
--secondary100: #f4fdff has a strong blue tint, breaking what should be a pure neutral ramp. The secondary scale (near-white → #9d9d9d) also never reaches a true dark shade. Only 15 total uses.
🟡 P2 — Orphaned Standalone Tokens
7 tokens with zero usage: --text-label-white, --text-label-gray, --text-label-black, --purple-text, --purple-bg, --orange-text, --orange-bg.
🟡 P2 — Conflicting @layer Blocks
Three separate @layer base blocks exist in globals.css. Block #2 (lines 323–330) is dead code, fully superseded by Block #3 (lines 332–339). Creates confusion and bloat.
⚪ P2 — 385 Hardcoded Hex Values
~385 hardcoded hex colors in .tsx files (170+ unique values). Worst: CategoryModal.tsx (32 hardcoded colors), PartnershipGrid.tsx (21), CategoryComboBox.tsx (14). Untokenized and impossible to update systematically.
Impact summary: P0 bugs mean three core Tailwind utility classes (bg-error, bg-warning, bg-info) produce no styles in production. The 27 toast calls using toast.error() and toast.warning() render incorrectly because the underlying tokens are broken.
02 · Architecture

Two-Tier Color Architecture

Every color in VendHub flows through two layers. Designers and developers should only ever reach for semantic tokens — never primitives directly.

🎨
Color Flow
Layer 0
Figma Variables
Color styles defined in the design file. Single source of truth for designers.
→
Layer 1
CSS Primitives
Raw palette variables in :root. Never used directly in components.
→
Layer 2
Semantic Tokens
Purpose-named tokens like --text-default, --bg-surface. What components consume.
Layer 1 — Primitive Palette
Raw color values organized by hue family. Six palettes: Neutral, Primary (brand green), Success (emerald), Warning (amber), Error (red), Info (blue). Each palette has 5–11 stops.

Rule: These exist to give the semantic layer something to reference. They should never appear in component code.
Layer 2 — Semantic Tokens
Named by purpose, not appearance. Grouped into five categories: Text (what to render), Background (surfaces), Border (edges/dividers), Interactive (brand actions), Status (success/warning/error/info).

Rule: Every component and page uses only these tokens.
Cross-Reference — What World-Class Systems Do
SystemPrimitive LayerSemantic LayerKey Insight Applied
Material Design 3 Tonal palettes (0–100) Color roles: primary, on-primary, container, on-container Every bg token has a paired "on-" foreground token
GitHub Primer Scale tokens (fg.*, canvas.*) fg.default / fg.muted / fg.subtle / fg.onEmphasis 4-level text hierarchy (default → muted → subtle → on-emphasis)
Shopify Polaris v11 Color scale --p-color-bg-surface, --p-color-text, --p-color-border Status colors use 4 slots: bg / border / text / icon
IBM Carbon 10-step hue scales layer-01/02/03, text-primary/secondary/disabled Layered background system (nested surfaces get darker)
Vercel Geist Neutral ramp bg-[value], text-[value], border-[value] Minimal: default / muted / subtle / accent — easy to learn
VendHub v2 approach: Adopts Polaris's 4-slot status pattern + Primer's 4-level text hierarchy + Material's "on-color" foreground pairing. Simpler than Carbon, more thorough than Geist.

Focus Ring Standard
All interactive elements must use --border-focus for keyboard navigation. Implementation in Tailwind: focus-visible:ring-2 focus-visible:ring-primary-600 focus-visible:ring-offset-2. Never suppress focus outlines — they are a legal accessibility requirement.
Status Borders
Form fields with status (error/warning/success) should use the status-specific border token, not these neutral ones. Use --error-border for error inputs, --success-border for confirmed inputs. The neutral borders cover default and hover states only.

10 · globals.css

globals.css — What to Fix

Seven targeted changes fix all P0 and P1 bugs. Apply in order. Each change is isolated — no cascading side effects when done correctly.

1
Fix shadcn --primary HSL conflict
Replace the existing --primary token in the shadcn block. The old value (162 43% 34%) resolves to teal. The new value must match brand green #16A34A.
/* BEFORE — teal (WRONG) */ --primary: 162 43% 34%; /* AFTER — brand green ✓ */ --primary: 142 72% 29%; /* hsl(142,72%,29%) = #16A34A */ --primary-foreground: 0 0% 100%; /* white text on primary buttons */
2
Fix --success syntax + foreground bug
HSL values must use spaces, not commas (modern CSS). Also fix --success-foreground which was set to the same value as --success.
/* BEFORE — broken */ --success: 143, 35%, 44%; /* commas = invalid */ --success-foreground: 143, 35%, 44%; /* same as success = green-on-green */ /* AFTER ✓ */ --success: 160 84% 20%; /* hsl(160,84%,20%) ≈ #059669 */ --success-foreground: 0 0% 100%; /* white ✓ */
3
Define the 6 missing tokens
--error, --error-foreground, --warning, --warning-foreground, --info, --info-foreground are referenced in tailwind.config but never defined. Add these to :root:
/* Add to :root — fixes bg-error, bg-warning, bg-info */ --error: 0 72% 51%; /* hsl = #DC2626 */ --error-foreground: 0 0% 100%; /* white */ --warning: 32 95% 44%; /* hsl = #D97706 */ --warning-foreground:0 0% 100%; /* white */ --info: 221 83% 53%; /* hsl = #2563EB */ --info-foreground: 0 0% 100%; /* white */
4
Add semantic token block
Add the full semantic token layer below the primitive palette block in :root. These power the new token system:
/* ── SEMANTIC TEXT ─────────────── */ --text-default: var(--neutral-900); /* #111827 */ --text-secondary: var(--neutral-600); /* #4B5563 */ --text-muted: var(--neutral-500); /* #6B7280 */ --text-disabled: var(--neutral-400); /* #9CA3AF */ --text-inverse: #ffffff; --text-link: var(--primary-600); /* #16A34A */ --text-brand: var(--primary-600); /* ── SEMANTIC BACKGROUND ────────── */ --bg-default: #ffffff; --bg-subtle: var(--neutral-50); /* #F9FAFB */ --bg-surface: var(--neutral-100); /* #F3F4F6 */ --bg-muted: var(--neutral-200); /* #E5E7EB */ --bg-brand: var(--primary-600); --bg-brand-subtle: var(--primary-50); --bg-overlay: rgba(0,0,0,0.5); /* ── SEMANTIC BORDER ────────────── */ --border-subtle: var(--neutral-100); --border-default: var(--neutral-200); --border-strong: var(--neutral-300); --border-focus: var(--primary-600); --border-brand: var(--primary-600); /* ── STATUS TOKENS ──────────────── */ --success-bg: var(--success-50); --success-border: var(--success-200); --success-text: var(--success-700); --success-icon: var(--success-500); --success-action: var(--success-600); --success-on: #ffffff; --warning-bg: var(--warning-50); --warning-border: var(--warning-200); --warning-text: var(--warning-700); --warning-icon: var(--warning-500); --warning-action: var(--warning-700); --warning-on: #ffffff; --error-bg: var(--error-50); --error-border: var(--error-200); --error-text: var(--error-700); --error-icon: var(--error-500); --error-action: var(--error-600); --error-on: #ffffff; --info-bg: var(--info-50); --info-border: var(--info-200); --info-text: var(--info-700); --info-icon: var(--info-500); --info-action: var(--info-600); --info-on: #ffffff;
5
Remove / deprecate orphaned tokens
Delete these from globals.css — all have zero usage: --primary250 (duplicate of primary50), entire --tertiary-* scale (13 tokens, zero usage), --warning850, --text-label-white/gray/black, --purple-text, --purple-bg, --orange-text, --orange-bg. Also remove the dead @layer base Block #2 (lines ~323–330).
6
Rename secondary-* → neutral-*
The --secondary-* tokens (15 total uses) are neutral grays that duplicate the typography scale. Rename to --neutral-* and update the 15 usages. Fix the blue-tinted --secondary100: replace with #F3F4F6.
7
Fix .animate-fade-in (bonus)
While in globals.css, fix the broken keyframe: animate-fade-in in tailwind.config.ts references a fadeIn keyframe that doesn't exist. Add it to globals.css: @keyframes fadeIn { from { opacity:0 } to { opacity:1 } }

15 · Dark Mode

Dark Mode Strategy

The app is currently light-mode only in practice — dark mode only overrides shadcn's 12 tokens, leaving 100+ custom tokens unchanged. Here's the minimum viable dark mode approach.

Current reality: The dark mode block in globals.css (.dark { ... }) only overrides shadcn tokens. When a user enables dark mode, all custom backgrounds, borders, and text remain their light-mode values. This is confusing and inconsistent. Fix in two phases.
Phase 1 — Minimum Viable (Now)
Override only the semantic layer tokens in the .dark block. Since all components should be using semantic tokens (after migration), this single block gives you a full dark mode:
/* globals.css */ .dark { /* Text */ --text-default: #F9FAFB; --text-secondary: #D1D5DB; --text-muted: #9CA3AF; --text-disabled: #6B7280; /* Backgrounds */ --bg-default: #111827; --bg-subtle: #1F2937; --bg-surface: #374151; --bg-muted: #4B5563; /* Borders */ --border-subtle: #1F2937; --border-default: #374151; --border-strong: #4B5563; }
Phase 2 — Full Dark Mode (Future)
After the token migration is complete (all raw Tailwind replaced with semantic tokens), dark mode becomes a single-file change — only globals.css needs updating.

Pre-requisite: Migrate ~6,400 raw Tailwind color uses to semantic tokens. Until then, dark mode will be incomplete even with the Phase 1 block in place.

Brand/status colors in dark mode should use lighter stops: --primary-400 (#4ADE80) on dark surfaces passes better contrast than primary-600.

16 · Do's & Don'ts

Color Usage Rules

Common mistakes to avoid, based on the 10 bugs found in the audit and patterns observed across 8,000+ color uses in the codebase.

✓ Do
Use --text-default for all primary text content. It maps to neutral-900 in light mode and automatically inverts in dark mode when the semantic token is overridden.
✗ Don't
Don't use text-gray-900 directly in new code. It bypasses the token system and will not respond to dark mode overrides or future brand changes.
✓ Do
Use the 6-slot pattern for status components: bg-error-bg border-error-border text-error-text. The slots are designed to always produce accessible combinations.
✗ Don't
Don't mix status families: bg-error-50 text-warning-700. Mixing slots from different statuses produces visually incoherent and potentially confusing results.
✓ Do
For warning components, always use text-warning-text (warning-700, #B45309) for text — not text-warning-600. The -600 stop fails WCAG AA on the warning-50 background.
✗ Don't
Don't use white text on bg-warning-500 (#F59E0B) or bg-warning-600 (#D97706). Both fail WCAG AA contrast for normal text (3.1:1 and 2.7:1 respectively).
✓ Do
Hardcode hex values only in CSS variable definitions in globals.css. Everywhere else — JSX, TSX, Tailwind classes — use tokens exclusively.
✗ Don't
Don't add inline styles with hex colors: style={{ color: "#16A34A" }}. There are already 385 of these in the codebase. Don't add more. Use className="text-primary-600".
✓ Do
For icons, use the -icon slot (text-success-icon) when the icon accompanies a status message. This uses the medium-weight stop (500) which is more visually balanced than the action stop (600).
✗ Don't
Don't reach for primitive palette values when a semantic token exists. If you're writing bg-green-100 for a success state, use bg-success-50 instead — it communicates intent and enables theming.

18 · Governance

Color System Governance

Rules for maintaining the system as it evolves. Prevents re-accumulating the token debt that led to the 10 audit findings.

✅ Approved Practices
✓
Use only semantic tokens (--text-*, --bg-*, --border-*, status slots) in all component code.
✓
Access primitive values (primary-600, neutral-500) directly in Tailwind when no semantic token fits the use case.
✓
Hardcode hex values only inside :root in globals.css — nowhere else in the codebase.
✓
Use the 6-slot status pattern for all alert, toast, badge, and inline feedback components.
✓
Include focus-visible:ring-2 ring-primary-600 on every interactive element for keyboard accessibility.
🚫 Banned Practices
✗
No style={{ color: "#hex" }} or style={{ backgroundColor: "#hex" }} in .tsx/.jsx files.
✗
No new raw Tailwind color classes (text-gray-*, bg-red-*, etc.) in new code. Migrate existing ones opportunistically.
✗
No new CSS custom properties added to globals.css without a documented use case and at least 3 component usages.
✗
No mixing status color families (no bg-error-bg text-warning-text).
✗
No white text on warning-500 or warning-600 backgrounds — contrast fails WCAG AA.
PR Checklist — Color Changes
✓
No new hardcoded hex values in any .tsx, .jsx, .ts, .js file. Run grep -r "#[0-9a-fA-F]\{3,6\}" --include="*.tsx" to verify.
✓
All new status UI (alerts, toasts, banners, badges) uses the 6-slot pattern from Section 03.
✓
Contrast verified for any new text-on-background combination using the contrast ratios in Section 06 of Part 1.
✓
New CSS tokens (if any) are documented in the Figma design system file before being added to globals.css.
✓
Focus rings present on all new interactive elements — inspect in keyboard navigation mode before merge.
When to Add a New Color Token
ScenarioAction
Need a color not in the palette (e.g. purple for a new feature) Add a full 5-stop primitive scale (50/200/500/600/700) + semantic slots. Document in Figma first. Min 3 component usages to justify.
Need a new shade of an existing palette Check if an existing stop satisfies the need. If genuinely a new stop is needed, add it to the primitive scale only — then use it via a semantic token, never directly.
Need a one-off color for a specific feature (e.g. banner builder) Use an isolated, well-named token block (like the existing banner-builder palette). Never add these to the main semantic layer.
A component needs a color that doesn't match any semantic role Re-examine the semantic intent. If genuinely novel, propose a new semantic token in the design system file first, get designer + lead dev review before adding to globals.css.
