01 · Grid System

Three-Tier Grid System

VendHub uses a responsive column grid that shifts across three breakpoints — 12 columns on desktop, 8 on tablet, 4 on mobile. The grid is applied to the content area frame, never the full viewport or artboard.

Critical rule: The grid is applied to the content area frame only — not the full 1440px artboard. In admin layouts, the grid starts after the sidebar. Column widths (e.g., 84.67px) are Figma display values only — never written in code. Browsers handle sub-pixel rendering automatically with grid-template-columns: repeat(12, 1fr).

03 · Column Spans

Column Span Usage Guide

Content must use standard column spans — never arbitrary widths. Desktop uses spans 3, 4, 6, 12. Tablet uses 2, 4, 8. Mobile uses 2, 4.

2 — cols 01–02 cols 01–02 Small cards (2 per row mobile), compact widgets
3 cols 01–03 — — Stat widgets, small tiles, product tiles (4 per row)
4 cols 01–04 cols 01–04 (4/4 split) span full Balanced card layouts, 3-per-row grids, full mobile content

⚠️
Product Grid — Documented Exception to the 12-Column System

Product cards use a fixed nested grid inside the container — not column spans. This is intentional and follows the same pattern used by Amazon, Shopify, and most e-commerce platforms.

12 · PageContainer Component
Shared layout wrapper
A single component that enforces consistent max-width, horizontal padding, and page-level spacing across all admin and customer-facing pages. Replace ad-hoc div className="max-w-… mx-auto px-…" patterns with this.
components/layout/page-container.tsx
import { cn } from '@/lib/utils' type ContainerVariant = 'admin' | 'customer' | 'form' | 'full' interface PageContainerProps { children: React.ReactNode variant?: ContainerVariant // default: 'admin' className?: string } const variantClasses: Record<ContainerVariant, string> = { // Admin — fluid, respects sidebar, standard page padding admin: 'w-full px-4 md:px-6 lg:px-8 py-6', // Customer — centered, 1280px cap, generous horizontal padding customer: 'mx-auto w-full max-w-content px-4 sm:px-6 xl:px-8 py-8', // Form — narrower centred column, ideal for checkout / settings form: 'mx-auto w-full max-w-prose px-4 sm:px-6 py-8', // Full — edge-to-edge, no horizontal padding (use for maps, dashboards) full: 'w-full', } export function PageContainer({ children, variant = 'admin', className, }: PageContainerProps) { return ( <div className={cn(variantClasses[variant], className)}> {children} </div> ) }
Usage examples
// Admin dashboard page <PageContainer> <DashboardContent /> </PageContainer> // Customer marketplace page <PageContainer variant="customer"> <MarketplaceGrid /> </PageContainer> // Settings / onboarding form <PageContainer variant="form"> <ProfileForm /> </PageContainer> // Extend with one-off overrides <PageContainer className="pt-0"> <SpecialPage /> </PageContainer>
ℹ️ Why four variants?
Each variant encodes a complete set of width + padding rules. Passing variant is safer than composing Tailwind classes ad-hoc because it prevents invalid combos like mixing max-w-content with fluid admin padding.
⚠️ Don't nest PageContainers
Only wrap the outermost content region of a page. Nesting two containers doubles the padding and can break max-width constraints. If you need inner capping, use a plain div with a max-w-prose class directly.

18 · Governance
Rules that prevent layout drift
Treat these as PR review checklist items. Any layout change should pass all applicable rules before merge.
✅ Required
1
Use PageContainer for every page
No page-level wrapper should duplicate max-width + padding logic inline.
2
Mobile-first column spans
Always write the mobile default first, then add sm:, md:, lg: overrides. Never omit the base (no-prefix) value.
3
Use approved column spans only
Desktop: 3, 4, 6, 12. Tablet: 2, 4, 8. Mobile: 2, 4. Any other span needs design approval.
4
Use gap-4 / gap-5 / gap-6
16px mobile, 20px tablet, 24px desktop grid gutters — always via Tailwind gap class, never margin hacks.
5
Use design token max-widths
max-w-page, max-w-content, max-w-prose. No arbitrary max-w-[*px] in layout wrappers.
🚫 Prohibited
1
No max-w-8xl
Undefined — generates no CSS. Use max-w-page instead (1280px).
2
No magic number padding in layout wrappers
No px-[32px], py-[20px], or style={{ padding: … }} at page/section level.
3
No xs: breakpoint in new code
The xs: prefix exists only for legacy support. All new responsive code starts at sm:.
4
No hard-coded sidebar widths
Use w-sidebar-full (256px) or w-sidebar-rail (64px) tokens, not w-[256px] literals scattered across files.
5
Never nest PageContainers
One PageContainer per page root. Inner content regions use a plain div with specific width utilities if needed.
ℹ️ PR review checklist
For any PR touching layout files, verify: (1) no max-w-8xl, (2) no arbitrary bracket padding at page level, (3) PageContainer used, (4) column spans are in the approved set, (5) mobile breakpoint is always the base (no-prefix) class.

19 · Layout Components
Stack, Inline & Grid Layout Components
Addresses audit finding P2: "No shared Stack, Inline, or PageWrapper components." These enforce consistent spacing without manual utility class assembly. Inspired by Shopify Polaris (<BlockStack>, <InlineGrid>) and GitHub Primer (<Stack>).
Why components, not just utility classes?
Utility classes like space-y-6 work but can drift — one developer uses space-y-4, another gap-8. Components enforce the spacing token through props with a constrained API. Use these for all new page layouts. Existing pages can migrate incrementally.
Stack — vertical spacing between sections
// components/layout/stack.tsx import { cn } from "@/lib/utils" const gapMap = { 4: "gap-4", // 16px — compact 5: "gap-5", // 20px — form fields 6: "gap-6", // 24px — default section gap 8: "gap-8", // 32px — between major sections 12: "gap-12", // 48px — page-level separation } as const interface StackProps extends React.HTMLAttributes<HTMLDivElement> { gap?: keyof typeof gapMap } export function Stack({ gap = 6, className, children, ...props }: StackProps) { return ( <div className={cn("flex flex-col", gapMap[gap], className)} {...props}> {children} </div> ) } // Usage: // <Stack gap={6}> ← 24px gap (default) // <Heading level={1}>Dashboard</Heading> // <StatsGrid /> // <RecentOrders /> // </Stack>
Inline — horizontal layout with responsive wrapping
// components/layout/inline.tsx import { cn } from "@/lib/utils" interface InlineProps extends React.HTMLAttributes<HTMLDivElement> { gap?: 2 | 3 | 4 | 6 | 8 align?: "start" | "center" | "end" | "between" wrap?: boolean } export function Inline({ gap = 4, align = "start", wrap = false, className, children, ...props }: InlineProps) { return ( <div className={cn( "flex", `gap-${gap}`, align === "between" ? "justify-between" : `items-${align}`, wrap && "flex-wrap", className )} {...props}>{children}</div> ) } // Usage: // <Inline gap={4} align="between"> // <Heading level={2}>Orders</Heading> // <Button>Export</Button> // </Inline>
ResponsiveGrid — the three-tier grid as a component
// components/layout/responsive-grid.tsx import { cn } from "@/lib/utils" interface ResponsiveGridProps extends React.HTMLAttributes<HTMLDivElement> { cols?: { mobile?: 1|2|4, tablet?: 2|4|8, desktop?: 3|4|6|12 } } export function ResponsiveGrid({ cols = { mobile: 1, tablet: 2, desktop: 3 }, className, children, ...props }: ResponsiveGridProps) { return ( <div className={cn( "grid gap-4 md:gap-5 lg:gap-6", `grid-cols-${cols.mobile}`, `md:grid-cols-${cols.tablet}`, `lg:grid-cols-${cols.desktop}`, className )} {...props}>{children}</div> ) } // Usage: // <ResponsiveGrid cols={{ mobile: 1, tablet: 2, desktop: 3 }}> // <StatCard /> <StatCard /> <StatCard /> // </ResponsiveGrid>
Gradual adoption
Use these components in all new code. For existing files, migrate when touching a file for other reasons (bug fixes, features). The utility class patterns from Section 13 still work — these are an ergonomic layer on top.
Why only 5 gap values in Stack?
The constrained gap prop (4/5/6/8/12) maps to the spacing system's component and layout tiers. This prevents developers from picking arbitrary values. If a new gap is genuinely needed, it must go through a design token PR first.
