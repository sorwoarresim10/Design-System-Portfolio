01 · Philosophy

Elevation-Based, Semantic, Perceptually Soft

Every shadow token represents a fixed position in the UI elevation hierarchy. Tailwind's default shadows (shadow-sm, shadow-lg) are designed for documentation, not production UIs.

No Tailwind defaults
Tailwind's built-in shadow scale (shadow, shadow-md, shadow-xl) has no semantic meaning — developers guess which to reach for. These tokens replace them entirely with names that describe what the shadow is for.
Dual-layer construction
Higher levels use two shadow layers: a diffuse ambient layer (large radius, low opacity) plus a sharp key layer (tight radius, slightly higher opacity). This mirrors real-world light, making elevation feel physically grounded.
Always transition on hover
Any component that changes shadow on interaction must include transition-shadow duration-200. Without it, the shadow snaps instantly and feels broken. This is a hard rule, not a suggestion.
Alignment with spacing system: The shadow hierarchy mirrors the spacing system's tier structure. Components at the same density tier share the same shadow level — flat sections use none, page cards use card, floating panels use raised or dropdown, and elevated overlays use modal.

03 · Tailwind Config

Token Definitions

Drop into tailwind.config.ts under theme.extend.boxShadow. Replaces all Tailwind default shadows.

// tailwind.config.ts → theme.extend.boxShadow boxShadow: { // L0 — absolutely flat (use Tailwind's built-in shadow-none) // L1 — page sections, sticky bars, subtle separators surface: "0 1px 2px rgba(0,0,0,0.04)", // L2 — default cards and UI blocks (dual-layer) card: "0 1px 3px rgba(0,0,0,0.06), 0 1px 2px rgba(0,0,0,0.04)", // L2 hover — interaction feedback on cards "card-hover": "0 4px 8px rgba(0,0,0,0.08)", // L3 — raised panels, sidebars, nested cards, in-page panels raised: "0 4px 12px rgba(0,0,0,0.06), 0 2px 4px rgba(0,0,0,0.04)", // L4 — floating UI: dropdowns, selects, tooltips, popovers dropdown: "0 8px 16px rgba(0,0,0,0.10), 0 2px 6px rgba(0,0,0,0.04)", // L5 — highest elevation: modals, dialogs, sheets, command palette modal: "0 20px 40px rgba(0,0,0,0.12), 0 8px 16px rgba(0,0,0,0.06)", },
Naming change: The old "L2.5" is now L3 (raised). Old L3 (dropdown) is now L4. Old L4 (modal) is now L5. The token names (shadow-raised, shadow-dropdown, shadow-modal) are unchanged — only the level numbers shifted. Code using semantic token names needs zero changes.
04 · L3 Raised

Why "Raised" Is a First-Class Level

The gap between shadow-card (3px/6%) and shadow-dropdown (16px/10%) was too large — a full 13px spread jump with no intermediate step. Several VendHub components lived in this gap and were being misassigned to Tailwind defaults.

L3 shadow-raised
"0 4px 12px rgba(0,0,0,0.06), 0 2px 4px rgba(0,0,0,0.04)"
Sits between card (3px) and dropdown (16px). The diffuse layer at 12px spread creates clear visual separation without the full floating appearance of a dropdown. Dual-layer construction grounds the component while the increased spread signals additional prominence.
Why the gap existed — and hurt
The original system had L2 (card) jump directly to L3 (dropdown) — a 13px spread increase. Components like BuyerRouteCard sidebar, admin content panels, and nested cards couldn't sit at L2 (too subtle) but didn't belong at dropdown (too heavy). Developers reached for shadow-lg from Tailwind's defaults, bypassing the token system entirely.

Optical justification: The L3 shadow (0 4px 12px rgba(0,0,0,0.06)) is optically positioned at the midpoint of the L2→L4 perceptual range, not the mathematical midpoint. Shadow perception is logarithmic — a 6px spread looks far closer to 3px than to 16px. The 12px value was chosen to land at approximately equal perceptual distance from both neighbors.

07 · Industry Reference

How Many Levels Are Ideal?

The optimal number is 5–6 semantic levels. This is the consensus across major design systems.

Material Design 3
0 — flat1 — tonal2 — card3 — floating4 — nav drawer5 — dialog
6 levels (0–5). Google names each level after its primary use case, not pixel values. VendHub directly mirrors this philosophy.
Linear App
0 — none1 — card2 — overlay3 — modal
4 levels. Extremely soft. Linear's flat-first UI needs fewer levels. Fewer = more intentional.
Stripe
xs — surfacesm — cardmd — dropdownlg — modalxl — overlay
5 levels. Size-named tokens with documented semantic intent. Clear escalation from subtle surface to full overlay.
◉
GitHub Primer
restingraisedoverlayfloatingdialog
5 levels, named by role. GitHub's "raised" is directly analogous to VendHub's L3 — a level for panels that are elevated but not floating.
V
Vercel
smallmediumlargeextra-large
4 levels, ultra-soft. Vercel's dark-first interface needs fewer, subtler shadows. Size tokens with semantic docs.
V
VendHub (this system)
L0 — noneL1 — surfaceL2 — cardL3 — raisedL4 — dropdownL5 — modal
6 levels (0–5). Aligns with Material Design 3's count. Clean integer numbering. The L3 (raised) level fills the gap that GitHub Primer also fills with their "raised" token.
Advisory: When to stop adding levels
A new level is justified only when a real component cannot be correctly assigned to any existing level, AND when that component is perceptually distinct from its nearest neighbors. VendHub's L3 passes both tests. Adding a 7th level (e.g. between dropdown and modal) would not pass the second test — the jump from 16px to 40px maps cleanly to "floating vs. blocking." Do not add levels without concrete justification — more levels create decision fatigue, not precision.

09 · Philosophy

Semantic, Role-Based Curvature

Radius is assigned by component role, not by size or aesthetics. Every developer has one answer to "what radius does this component use?" — zero guessing between rounded-md, rounded-lg, and rounded-xl.

Role, not size
A button and an input share the same radius because they occupy the same semantic tier — interactive controls. Size alone should never determine curvature.
Strict hierarchy
The hierarchy enforces: control (6px) < container (8px) < surface (12px). A form field is never rounder than its containing card. A card is never rounder than its modal.
One token per component type
Every component type maps to exactly one token. There are no per-context overrides — a dropdown always uses rounded-container, never sometimes rounded-lg and sometimes rounded-xl.

10 · Token Definitions

Five Radius Tokens

All Tailwind defaults (rounded-md, rounded-lg, rounded-xl) are retired from use in favor of these named values.

TokenTailwind ClassValueRoleComponents
inline rounded-inline 4px Inline elements Code chips, micro-tags, inline annotations
control rounded-control 6px Interactive controls Buttons, inputs, selects, checkboxes, textareas
container rounded-container 8px UI containers Cards, panels, dropdowns, popovers, tooltips
surface rounded-surface 12px Elevated surfaces Modals, dialogs, sheets, drawer panels
pill rounded-pill 9999px Fully rounded Badges, tags, status indicators, progress bars, tabs

13 · Rules

Do's & Don'ts

DO
✓ Use semantic tokens only — never rounded-md or raw px
✓ Enforce hierarchy: control < container < surface
✓ Same component type → same token, always
✓ Admin sidebar: single radius for all nav levels (container)
DON'T
✗ Mix rounded-md, rounded-lg, rounded-xl at random
✗ Use arbitrary values like rounded-[10px]
✗ Use multiple radii inside the same component group
✗ Override hierarchy — a form field must not be rounder than its modal
