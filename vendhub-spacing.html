01 · Problems

12 Spacing Issues Found in the Codebase

Audit of ~8,000 spacing-related utility uses across the app revealed systemic inconsistency. Zero shared patterns between features built at different times.

🔴 P0 — max-w-8xl is a no-op
md:max-w-8xl appears in 3 critical page layouts (Dashboard, Routes, Leads) but is never defined in tailwind.config.ts. It silently produces no CSS. Pages appear full-width with only mx-20 constraining them.
🟠 P1 — 6 Different Page Padding Patterns
Every feature uses a different horizontal padding strategy:
px-8 · px-4 sm:px-6 lg:px-8 · px-4 sm:px-6 lg:px-10 · px-4 md:px-8 lg:px-12 · px-4 sm:px-6 (missing lg) · no padding at all (Dashboard, Leads, Partners).
🟠 P1 — Sibling Pages With Different Top Padding
Within the same insights section (same layout wrapper): pt-20, pt-16, pt-28. Three sibling pages, three different values, no documented reason.
🟠 P1 — 6+ Card Padding Patterns
p-4 · p-4 md:p-6 · p-5 · p-6 · p-6 md:p-10 · p-10 · p-4 sm:p-6 md:p-8. No single source of truth for what a card should feel like.
🟠 P1 — No Consistent Max-Width Strategy
9 different max-width approaches across features: container max-w-7xl, max-w-[93rem], max-w-none, max-w-4xl, no max-width at all, and the broken max-w-8xl.
🟠 P1 — Admin Pages With Only 8px Padding
Admin content areas use only p-2 (8px) total padding — content is nearly flush with the viewport edge. No responsive expansion.
🟡 P2 — Magic Number: pt-[62px]
USG layout uses pt-[62px] as a header offset. The header is actually 64px (space.16). This 2px discrepancy is silent and visually creates a gap. Should be pt-16 (64px).
🟡 P2 — Hardcoded calc() Heights
max-h-[calc(100vh-50px)], h-[calc(100vh-200px)] — arbitrary subtracted values. These break when the header size changes. Should use CSS variable var(--header-h).
🟡 P2 — Arbitrary Fixed Footer Offsets
pb-24 (Routes Explore) and pb-20 (Financing) are magic-number bottom paddings accounting for a fixed footer bar. These values aren't shared or documented.
🟡 P2 — Arbitrary Card Width
Onboarding uses md:w-[553px] — a completely arbitrary card width with no semantic meaning. Should map to a standard container token (max-w-md = 448px or max-w-lg = 512px).
🟡 P2 — Route Detail Uses 1488px Container
max-w-[93rem] (1488px) is used in Route Detail as a one-off. No other page uses this. Exceeds the standard page max-width with no documented rationale.
⚪ P2 — No Spacing Components / Utilities
No shared <Stack>, <Inline>, or <PageWrapper> components. Every file assembles its own spacing from scratch, leading to all of the above.
Root cause: No spacing token system existed. Every developer chose their own values. The fix requires (1) defining canonical tokens, (2) a page layout wrapper component, and (3) a card padding standard — addressing ~80% of all inconsistencies.
02 · Philosophy

The 4px Base Unit

All spacing in VendHub is a multiple of 4px. This single constraint produces visual harmony across the entire interface — every gap, padding, and margin aligns to an invisible 4px grid.

Why 4px?
4px (0.25rem) is the most widely adopted base unit in design systems. It's small enough to provide fine control at the micro level (icons, text gaps) while scaling naturally to layout sizes (64px = 16 steps).

At standard display density, 4px is visually indistinguishable from 3px or 5px — making it a practical minimum. Below 4px, use border widths or line-heights instead.
The Three Groups Rule
Every spacing token belongs to one of three groups based on what it spaces:

Micro (0–12px) — spacing within UI elements (icon-to-text, label-to-input)
Component (16–24px) — spacing inside components (padding, gaps between fields)
Layout (32–80px) — spacing between components and page sections

If your spacing crosses a group boundary, something is likely wrong.
Cross-Reference — World-Class Design Systems
SystemBase UnitScale RangeToken CountKey Insight Applied to VendHub
Material Design 3 4dp 4–64dp ~12 tokens Spacing assigned by role, not by eye. Component padding ≠ layout gap.
GitHub Primer 4px 0–128px 16 tokens Named tokens map 1:1 to Tailwind's built-in scale (spacer-4 = p-4)

Key insight: VendHub's 12-token spacing scale (space.0 through space.20) maps perfectly to Tailwind's built-in spacing scale (p-0 through p-20). No custom Tailwind config is needed for the core scale. Only semantic aliases (like --header-h: 64px) need to be added.

10 · Tailwind Config

tailwind.config.ts — Spacing Setup

VendHub's spacing scale maps exactly to Tailwind's built-in scale — no custom values needed for core tokens. Only semantic aliases require additions.

Important: Because space.1 = p-1 = 4px, space.2 = p-2 = 8px, etc., the core scale is already available in Tailwind with zero config. The additions below add semantic aliases (--header-h, max-w-page) and fix the broken max-w-8xl no-op.
// tailwind.config.ts — theme.extend additions for spacing extend: { // ── SEMANTIC SPACING ALIASES ───────────────────────────── // These add intent-named classes on top of the default scale spacing: { "header-h": "64px", /* h-header-h, pt-header-h, mt-header-h */ "page-pad": "32px", /* px-page-pad — standard page H-padding */ "page-pad-sm": "16px", /* mobile page padding */ "card-pad": "24px", /* p-card-pad — default card padding */ "card-pad-sm": "16px", /* p-card-pad-sm — compact card padding */ "section-gap": "48px", /* mb-section-gap — between page sections */ "card-gap": "24px", /* gap-card-gap — gap between cards */ "footer-bar": "64px", /* pb-footer-bar — fixed bottom bar offset */ }, // ── MAX-WIDTH ADDITIONS ─────────────────────────────────── maxWidth: { "page": "1280px", /* max-w-page — standard page container */ "content": "768px", /* max-w-content — text/article column */ "form": "512px", /* max-w-form — standard form width */ "card": "384px", /* max-w-card — single column card width */ }, // ── CSS VARIABLES (for calc() and JS access) ───────────── // Add to globals.css :root block: // --header-h: 64px; → use in calc(100vh - var(--header-h)) // --card-pad: 24px; → use in CSS-in-JS if needed }, // ── REMOVE — these were broken no-ops ──────────────────── // max-w-8xl was never defined. Delete all usages: // grep -r "max-w-8xl" --include="*.tsx" → replace with max-w-page
Why semantic aliases?
pt-header-h communicates intent — "this is the header offset" — while pt-16 is ambiguous (is it a coincidence it's 64px, or intentional?). When the header height changes from 64px to 56px, you update one variable, not grep for every pt-16.
CSS variable for calc() patterns
Add to globals.css :root:
--header-h: 64px; --footer-bar-h: 64px;
Then use: min-h-[calc(100vh-var(--header-h))] instead of the current hardcoded calc(100vh-64px).

The one-step rule: Between any two breakpoints, spacing should jump by at most one step in the scale. px-4 lg:px-8 is fine (skips sm, 2 steps). px-2 lg:px-12 is too aggressive — 5 steps feels jarring on a narrow window.

15 · Do's & Don'ts

Spacing Rules

The most common spacing mistakes in the VendHub codebase, and how to fix them going forward.

✓ Do
Use scale tokens for all spacing. gap-4, p-6, mb-12. Every value should be a multiple of 4px from the defined scale.
✗ Don't
Don't use arbitrary pixel values: p-[18px], mt-[13px], gap-[7px]. If 16px isn't enough, use 20px — not 18px.
✓ Do
Use pt-header-h (or pt-16) for the fixed header offset. This makes it searchable and meaningful — not a magic number.
✗ Don't
Don't use pt-[62px] as a header offset. The header is 64px. 62px is wrong and the brackets make it invisible to systematic refactors.
✓ Do
Use min-h-[calc(100vh-var(--header-h))] for full-height content areas. The CSS variable makes the dependency explicit and updatable.
✗ Don't
Don't write min-h-[calc(100vh-64px)] or h-[calc(100vh-200px)]. Hardcoded subtracted values create silent coupling to layout decisions.
✓ Do
Use max-w-page mx-auto for the standard page container. All pages should have the same container width unless there's a documented reason.
✗ Don't
Don't use max-w-8xl — it's undefined and produces no CSS. Don't use max-w-[93rem] as a one-off. Standardize on max-w-page (1280px).
✓ Do
Vary spacing systematically across a page: cards use the same padding, sections use the same gap. Consistency within a level of the hierarchy.
✗ Don't
Don't mix card padding tiers on the same page. Cards in the same group should all use p-6 (default) or all use p-4 (compact) — never a mix of different values.

Spacing System Governance

Rules for maintaining the spacing system as the app evolves. Prevents re-accumulating the 12 inconsistencies found in the audit.

✅ Approved Practices
✓
Use only tokens from the 12-token scale for all spacing (p-0 through p-20 in multiples). Any value not in the scale needs explicit approval.
✓
Wrap every new page in <PageWrapper> — this guarantees correct horizontal padding, max-width, and vertical padding automatically.
✓
Use pt-header-h for the fixed header offset, pb-footer-bar for any fixed bottom bar offset. Never hardcode the px value.
✓
All cards within the same visual group must use the same padding token. Card padding variation is allowed between groups (compact vs featured), not within.
✓
Use gap-* (flexbox/grid gap) rather than mb-* on children where possible. Gap is more maintainable and doesn't require clearing on the last child.
🚫 Banned Practices
✗
No arbitrary pixel values in brackets: p-[18px], mt-[7px], gap-[13px]. If a value is needed, round to the nearest scale step.
✗
No max-w-8xl — it's undefined. No max-w-[Xrem] unless documented in this system with a named token.
✗
No calc(100vh - {hardcoded}px). Use calc(100vh - var(--header-h)) instead.
✗
No spacing inconsistency between sibling pages. If /insights/dashboard uses pt-20, then /insights/providers must too — or both should change to the standard.
✗
No new page-level spacing without using PageWrapper or explicitly documenting why a custom wrapper is needed.
PR Checklist — Spacing Changes
✓
No arbitrary values: Run grep -r "\[.*px\]" --include="*.tsx". Any new bracket-px should be justified in the PR description.
✓
New pages use PageWrapper: Every new page component wraps its content in <PageWrapper>.
✓
Card padding consistent: All cards on the same page use the same padding token.
✓
Sibling pages match: If you update spacing on one page, check its sibling pages in the same section and align them.
✓
No new max-w-8xl: Check that the undefined max-w-8xl class wasn't reintroduced. Use max-w-page instead.
