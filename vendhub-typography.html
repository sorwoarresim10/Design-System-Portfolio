01

Problems identified

7 root causes diagnosed from the full codebase audit. These are the exact issues the v2 system solves — understanding them first makes every design decision below obvious.

Scale & Hierarchy
P1 — Wrong body text size
text-sm (14px) is used as body text in 50.5% of cases. Standard SaaS uses 16px as body, 14px as secondary. Causes cramped interfaces and accessibility issues.
P2 — Heading hierarchy is broken
H3 can render larger than H1. Every heading level uses wildly different sizes per page with no enforced rule. An H3 at text-2xl beats an H1 at text-xl.
P3 — 51 arbitrary pixel sizes
text-[7px] text-[9px] text-[10px] text-[11px] text-[41px] text-[72px] break consistency and are impossible to update globally.
Weights & Colors
P4 — Unloaded font weights used
Plus Jakarta Sans loads 400/500/600/700. But font-light (300) is used 24× and font-extralight (200) twice. Browsers simulate → blurry, inconsistent rendering.
P5 — Split text color system
text-gray-* (2,193 uses) and text-typography-* (384 uses) coexist for the same purpose. Developers have no rule for which to reach for.
Architecture
P6 — No typography components
No shared <Heading> <Text> <Caption>. Every file manually assembles text-sm font-medium text-gray-500 combinations — 424 instances of the most common pattern alone.
P7 — Weight variation unsolvable in v1
The original 8 tokens embedded fontWeight inside fontSize tokens. When you added font-medium on top, two CSS rules fought over font-weight — unpredictable override order.
Cross-reference: How top design systems solve this
Material Design 3 (15 tokens), Shopify Polaris v11 (14 tokens), and GitHub Primer all separate size and weight into independent dimensions. They also give the same font size a different semantic name when the weight changes role — e.g. Material's "Title Medium" (16px/500) and "Body Large" (16px/400) are the same size, different weight, different name. VendHub v2 follows this exact pattern with the label token.
02

Architecture — 3-Layer System

Tokens are organized into three layers. Each layer has a clear rule about how weight works. Knowing which layer you're in tells you everything about how to use it.

Layer 1 — Display
Marketing, heroes, landing pages. Weight is always Bold (700). Never override. These sizes only appear in promotional context.
displayXL displayLG
Weight: 700 embedded · Never composable
Layer 2 — Heading
Page structure, section headers, sub-sections. Weight is always SemiBold or Bold (600/700). Semantic HTML anchor: H1/H2/H3.
pageTitle sectionTitle subheading
Weight: 700/600 embedded · Never composable
Layer 3 — Body, Label & Caption
Content, UI chrome, metadata. Split into three roles. Body + Caption are composable. Label always Medium.
body bodySm label labelSm caption
body/caption: composable · label: 500 embedded

Body Layer — Composable Weight

Two tokens with no embedded weight. Always pair with an explicit font-* utility. This solves the v1 conflict where embedded fontWeight: 400 fought against font-medium.

Developer rule — always be explicit about weight
Body tokens have no default font-weight in the Tailwind config. Always write text-body font-normal, text-bodySm font-medium etc. "No weight class" means the element inherits from its parent — that can be unpredictable.

Minimum size rule — 12px floor
The audit found 39 instances of sub-12px text (7px, 8px, 9px, 10px, 11px). These all violate WCAG minimum text size guidelines and cause blurry rendering on non-retina displays. Replace all sub-12px text with text-caption or redesign the component to need less text.

Font Weights & Loading

Plus Jakarta Sans loads exactly 4 weights. The system is designed around these 4 — no synthetic weights, no degraded rendering.

Weight Value Tailwind Class Used In Loaded?
Regular 400 font-normal body, bodySm, caption (default reading) ✓ Loaded
Medium 500 font-medium label, labelSm (embedded) · body/bodySm emphasis ✓ Loaded
SemiBold 600 font-semibold sectionTitle, subheading (embedded) · body strong ✓ Loaded
Bold 700 font-bold displayXL, displayLG, pageTitle (embedded) ✓ Loaded
Light (synthesized) 300 font-light Found 24× in codebase — all to be removed ✗ Not loaded
ExtraLight (synthesized) 200 font-extralight Found 2× in codebase — both to be removed ✗ Not loaded
Next.js font loading — globals or layout
// app/layout.tsx or app/globals.css // Load exactly: 400, 500, 600, 700 @import url('https://fonts.googleapis.com/css2? family=Plus+Jakarta+Sans: wght@400;500;600;700 &display=swap'); // OR via next/font (preferred): const jakarta = PlusJakartaSans({ subsets: ['latin'], weight: ['400', '500', '600', '700'], variable: '--font-plus-jakarta', });
Never use these classes
font-light font-extralight font-thin
These weights are not loaded. Browsers simulate them by algorithmically thinning the 400 weight — resulting in blurry, inconsistent text across different rendering engines.
Remove font-jakarta redundancy (78 instances)
The font is applied globally on <html> via the layout. The 78 explicit font-jakarta classes across the codebase are redundant — safe to remove.

Tailwind Config

Drop this into tailwind.config.ts under theme.extend. The key change from v1: Display and Heading tokens embed fontWeight, Body and Caption tokens do not.

// tailwind.config.ts — theme.extend.fontSize // 13 tokens · 3-layer system · v2 // All values in rem for browser-zoom accessibility (1rem = 16px default) fontSize: { // ── DISPLAY LAYER ────────────────────────────────────────── // fontWeight + letterSpacing embedded. Never add font-* on top. displayXL: ["4.5rem", { lineHeight: "5.375rem", fontWeight: "700", letterSpacing: "-0.02em" }], displayLG: ["3rem", { lineHeight: "3.625rem", fontWeight: "700", letterSpacing: "-0.015em" }], // ── HEADING LAYER ────────────────────────────────────────── // fontWeight + letterSpacing embedded. Use with H1/H2/H3 tags. pageTitle: ["2rem", { lineHeight: "2.5rem", fontWeight: "700", letterSpacing: "-0.01em" }], sectionTitle: ["1.5rem", { lineHeight: "2rem", fontWeight: "600", letterSpacing: "-0.005em" }], subheading: ["1.25rem",{ lineHeight: "1.75rem", fontWeight: "600", letterSpacing: "-0.005em" }], // ── BODY LAYER ───────────────────────────────────────────── // NO fontWeight. Always pair with font-normal / font-medium / font-semibold. body: ["1rem", { lineHeight: "1.5rem" }], bodySm: ["0.875rem",{ lineHeight: "1.25rem" }], // ── LABEL LAYER ──────────────────────────────────────────── // fontWeight: 500 embedded. Always Medium — UI chrome. label: ["0.875rem",{ lineHeight: "1.25rem", fontWeight: "500" }], labelSm: ["0.75rem", { lineHeight: "1rem", fontWeight: "500" }], // ── CAPTION LAYER ────────────────────────────────────────── // NO fontWeight. Composable. Usually font-normal, sometimes font-medium. caption: ["0.75rem", { lineHeight: "1rem" }], },
Why no fontWeight on body/caption?
In v1, bodySm: ["14px", { lineHeight: "20px", fontWeight: "400" }] generated .text-bodySm { font-weight: 400 }. When you added font-medium, both classes set font-weight and CSS specificity determined the winner unpredictably. Removing fontWeight from the config means the font-size token no longer sets weight at all — only your explicit font-* class does.
Tailwind v3 vs v4 note
VendHub uses Tailwind v3.4. In v4 (CSS-first config), the pattern moves to @theme blocks. The font-size values are identical — only the config format changes. Keep this file updated when upgrading.
Why rem instead of px?
All values use rem (relative to root font size) instead of px. When a user sets a custom base font size in their browser for accessibility, rem values scale proportionally while px values stay fixed. At default settings (16px root), 1rem = 16px — visually identical. Figma specs remain in px since Figma has no rem support. The conversion: px ÷ 16 = rem.
Why letterSpacing on Display/Heading only?
Negative letter-spacing (tracking) improves readability at large sizes by tightening the natural spacing of the typeface. At body sizes (14–16px) the default tracking is already optimal. The Figma text styles specify -1.5% to -0.2% for Display and Heading — these are now embedded in the config so developers get the same tracking as the Figma design without adding extra classes.
P0 · Font Loading Bug 26 usages of unloaded font weights in production

Plus Jakarta Sans currently loads only weights 400, 500, 600, 700. The production codebase uses font-light (weight 300) in 24 places and font-extralight (weight 200) in 2 places. Browsers synthesise missing weights — the result is blurry, inconsistent text that differs per browser and breaks on PDF export.

Current — broken (app/layout.tsx)
// Only 4 weights loaded — 300 & 200 are NOT here family=Plus+Jakarta+Sans:ital,wght@ 0,400;0,500;0,600;0,700;1,400
Fixed — two options (choose one)
// Option A: Add missing weights to the import family=Plus+Jakarta+Sans:ital,wght@ 0,300;0,400;0,500;0,600;0,700;1,400 // Option B (recommended): Remove all font-light/extralight // and replace with font-normal. Keeps bundle smaller. // grep -r "font-light\|font-extralight" --include="*.tsx"
Recommended approach (Option B): Replace all 26 usages with font-normal. The affected text is helper text, metadata, and captions — weights below 400 are rarely meaningful at small sizes. This avoids adding a network round-trip for 300/200 weights. Run: grep -rn "font-light\|font-extralight" app/ components/ --include="*.tsx" to find all 26 locations.

⚠️
Dual Text Color System — Audit Finding (79% bypass rate)

Production currently has two parallel text color systems in use simultaneously: text-gray-* (Tailwind default, 2,193 occurrences) and text-typography-* (design system tokens, 384 occurrences). This means 85% of text colors bypass the token system — a brand color change would require touching hundreds of files.

The migration target is text-neutral-*, not text-typography-*. The new Semantic color layer uses neutral-* naming (aligned with Tailwind) rather than the old typography-* custom scale. Both resolve to the same hex values. The migration table below maps the most common text-gray-* classes to their text-neutral-* equivalents — the values are identical, only the class name changes.

Do's & Don'ts

The 10 rules that prevent the most common mistakes.

DO
Always be explicit about weight on body/caption tokens. Write font-normal even when it's the default. Makes intent clear to every developer reading the code.
Use text-label for all form labels, table headers, nav items. It's always 14px Medium. You never need to think about weight.
Use semantic color tokens with typography. text-neutral-500 not text-gray-500.
Use text-labelSm for all status badges, chips, and tags. Single rule covers every status indicator in the app.
Apply heading tokens only to semantic heading elements. text-pageTitle goes on <h1>, not <div>.
DON'T
Don't manually assemble size + weight + color. Never write text-sm font-medium text-gray-500. That's exactly what the token system eliminates.
Don't add font-* to Display or Heading tokens. text-pageTitle font-bold is redundant — Bold is already embedded.
Don't use arbitrary pixel sizes. No text-[10px], text-[11px], text-[7px]. If it doesn't fit in 12px, redesign the component.
Don't use font-light or font-extralight. Those weights aren't loaded. Browsers synthesize them — blurry on screen, broken in PDF export.
Don't skip the explicit font-weight on body/caption tokens. Inherited weight from a parent component is unpredictable. Always be explicit.

Governance

Rules that prevent the system from degrading. The primary cause of the current mess is no review gate — any developer could add any text class anywhere. These rules fix that.

✓ Approved — always OK to use
Any of the 13 typography tokens from this system
font-normal font-medium font-semibold font-bold on composable tokens
Semantic color tokens: text-neutral-* text-primary-* text-error-* etc.
italic modifier on body/caption only (never on headings)
uppercase + tracking-wide on text-labelSm for all-caps labels
✗ Banned — never use
Raw Tailwind text sizes: text-xs text-sm text-base text-lg text-xl text-2xl etc.
Arbitrary sizes: text-[7px] text-[10px] text-[41px] etc.
Unloaded weights: font-light font-extralight font-thin
Raw Tailwind colors: text-gray-* text-green-* text-red-* etc.
Hardcoded hex in JSX: style={{ color: '#374151' }}
PR review checklist
1
Grep for text-sm text-xs text-base text-lg etc. — should be zero new instances.
2
Grep for text-gray- text-green- text-red- — should be zero new instances.
3
Every text-body / text-bodySm / text-caption must have an explicit font-* alongside it.
4
No font-light or font-extralight in any new file.
5
No arbitrary sizes. If a new size is genuinely needed, open a design token PR first.
When to add a new token
The 13-token system covers every VendHub pattern. A new token is justified only if:
① A genuine new UI pattern exists that none of the 13 tokens address (not just a different weight at an existing size).
② The pattern appears in at least 5 different places in the codebase — not a one-off special case.
③ A Figma text style for it has been created and reviewed in the design file first.
④ Both this document and the Figma file are updated in the same PR.

Responsive Typography

Display and Heading tokens scale down on smaller viewports. Body, Label, and Caption stay the same size across all breakpoints — they're already optimized for small screens.

Industry standard — every major system does this
Material Design 3 defines compact/medium/expanded size classes. Shopify Polaris scales tokens per breakpoint. GitHub Primer uses rem with responsive overrides. A 72px hero title on a 390px screen is unreadable without responsive scaling.
Token Desktop (lg+) Tablet (md) Mobile (default) Strategy
displayXL 72px / 86px 56px / 64px 40px / 48px Scales aggressively — hero text must fit mobile
displayLG 48px / 58px 40px / 48px 32px / 40px Landing sections scale to pageTitle size on mobile
pageTitle 32px / 40px 28px / 36px 24px / 32px Moderate scale — page titles must stay prominent
sectionTitle 24px / 32px 24px / 32px 20px / 28px Only scales at mobile — stable at tablet+
subheading 20px / 28px — no scaling Already small enough for mobile
body · bodySm · label · labelSm · caption No scaling — fixed across all breakpoints Body text ≤ 16px is readable at all sizes
Implementation — CSS custom properties approach
/* globals.css — responsive typography overrides */ /* Base = mobile-first (smallest sizes) */ :root { --font-display-xl: 2.5rem; /* 40px */ --lh-display-xl: 3rem; /* 48px */ --font-display-lg: 2rem; /* 32px */ --lh-display-lg: 2.5rem; /* 40px */ --font-page-title: 1.5rem; /* 24px */ --lh-page-title: 2rem; /* 32px */ --font-section-title: 1.25rem; /* 20px */ --lh-section-title: 1.75rem; /* 28px */ } @media (min-width: 768px) { :root { --font-display-xl: 3.5rem; /* 56px */ --lh-display-xl: 4rem; /* 64px */ --font-display-lg: 2.5rem; /* 40px */ --lh-display-lg: 3rem; /* 48px */ --font-page-title: 1.75rem; /* 28px */ --lh-page-title: 2.25rem; /* 36px */ --font-section-title: 1.5rem; /* 24px */ --lh-section-title: 2rem; /* 32px */ } } @media (min-width: 1024px) { :root { --font-display-xl: 4.5rem; /* 72px */ --lh-display-xl: 5.375rem; /* 86px */ --font-display-lg: 3rem; /* 48px */ --lh-display-lg: 3.625rem; /* 58px */ --font-page-title: 2rem; /* 32px */ --lh-page-title: 2.5rem; /* 40px */ --font-section-title: 1.5rem; /* 24px — same as tablet */ --lh-section-title: 2rem; /* 32px */ } }
Updated Tailwind config — using CSS variables
// tailwind.config.ts — responsive display/heading tokens // These reference CSS custom properties set in globals.css displayXL: ["var(--font-display-xl)", { lineHeight: "var(--lh-display-xl)", fontWeight: "700", letterSpacing: "-0.02em" }], displayLG: ["var(--font-display-lg)", { lineHeight: "var(--lh-display-lg)", fontWeight: "700", letterSpacing: "-0.015em" }], pageTitle: ["var(--font-page-title)", { lineHeight: "var(--lh-page-title)", fontWeight: "700", letterSpacing: "-0.01em" }], sectionTitle: ["var(--font-section-title)", { lineHeight: "var(--lh-section-title)", fontWeight: "600", letterSpacing: "-0.005em" }], // subheading, body, bodySm, label, labelSm, caption — keep static rem values
DO
Use text-displayXL and let the CSS variables handle scaling. You write one class — it responds to three breakpoints automatically.
DON'T
Don't manually override with text-displayXL md:text-displayLG lg:text-displayXL. The responsive scaling is built into the token itself via CSS variables.

Typography Components

Copy-paste TSX components that enforce the type system. These prevent drift by making it impossible to pick the wrong token — the component does it for you. Based on the same pattern used by Shopify Polaris (<Text variant="...">) and GitHub Primer (<Heading>).

Audit finding P6 — "No typography components"
The audit flagged zero shared <Heading> or <Text> components. Every file manually assembles utility class combos — 424 instances of text-sm font-medium alone. These components are the enforcement mechanism that turns governance rules from "please follow" to "can't violate."
Heading component — components/ui/heading.tsx
import { cva, type VariantProps } from "class-variance-authority" import { cn } from "@/lib/utils" const headingVariants = cva("text-neutral-900", { variants: { level: { 1: "text-pageTitle", 2: "text-sectionTitle", 3: "text-subheading", }, }, defaultVariants: { level: 2 }, }) type HeadingLevel = 1 | 2 | 3 interface HeadingProps extends React.HTMLAttributes<HTMLHeadingElement>, VariantProps<typeof headingVariants> { as?: `h${HeadingLevel}` } export function Heading({ level = 2, as, className, children, ...props }: HeadingProps) { const Tag = as ?? (`h${level}` as const) return <Tag className={cn(headingVariants({ level }), className)} {...props}> {children} </Tag> }
Text component — components/ui/text.tsx
import { cva, type VariantProps } from "class-variance-authority" import { cn } from "@/lib/utils" const textVariants = cva("", { variants: { variant: { "body": "text-body text-neutral-700", "bodySm": "text-bodySm text-neutral-600", "label": "text-label text-neutral-800", "labelSm": "text-labelSm", "caption": "text-caption text-neutral-500", }, weight: { "normal": "font-normal", "medium": "font-medium", "semibold": "font-semibold", }, }, defaultVariants: { variant: "bodySm", weight: "normal", }, }) interface TextProps extends React.HTMLAttributes<HTMLElement>, VariantProps<typeof textVariants> { as?: "p" | "span" | "div" | "label" | "th" } export function Text({ variant = "bodySm", weight = "normal", as: Tag = "span", className, children, ...props }: TextProps) { // Label/labelSm already embed Medium — skip weight prop const effectiveWeight = (variant === "label" || variant === "labelSm") ? undefined : weight return <Tag className={cn(textVariants({ variant, weight: effectiveWeight }), className)} {...props} > {children} </Tag> }
Usage examples
// Before — ad-hoc utility classes <h1 class="text-2xl font-bold text-gray-900"> Review Order </h1> <label class="text-sm font-medium text-gray-700"> Delivery Method </label> <span class="text-xs font-medium">Active</span> <p class="text-[10px] text-gray-400"> Updated 2h ago </p>
// After — enforced by components <Heading level={1}> Review Order </Heading> <Text variant="label" as="label"> Delivery Method </Text> <Text variant="labelSm">Active</Text> <Text variant="caption" as="p"> Updated 2h ago </Text>
Gradual adoption — not a big-bang rewrite
Use these components in all new code. For existing files, migrate page-by-page during the token migration (Step 17 in the Audit Action Plan). The utility class approach from the other sections still works — these components are an ergonomic layer on top, not a replacement for knowing the tokens.
Why CVA (class-variance-authority)?
CVA is already in VendHub's dependency tree via shadcn/ui. It provides type-safe variant props, auto-complete in VS Code, and composes cleanly with cn(). No new dependencies needed.
