01 · Philosophy

Why Named Z-Index Tokens

Without a z-index system, developers guess values. One component uses z-[999], another uses z-50, and they fight. VendHub's system assigns every layer a named token with gaps for future needs.

Deterministic
Every component knows exactly which layer it sits on. A dropdown (z-20) will always appear above a sticky header (z-10) and below a modal (z-40). No surprises.
Gapped
Tiers are spaced by +10 (0, 10, 20, 30, 40, 50). This leaves room for intermediate values (z-25 for popovers) without restructuring the entire scale.
Named, Not Numbered
Developers write z-modal not z-40. The name communicates intent — if the value changes, only the token updates. No grep required.
The one rule: Never write a raw z-index number in a component. Always use a named token. If a component needs a z-index not covered by the 9 tokens, document the case and add a new named token — don't use z-[999].

03 · Token Reference

Complete Token Table — 9 Layers

The definitive z-index scale for VendHub. Every component must use one of these named tokens.

TokenValueLayerTailwind ClassComponents
--z-base0 Base z-base Normal document flow. All content by default.
--z-raised1 Raised z-raised Cards on hover, floating action buttons within content.
--z-sticky10 Sticky z-sticky Main header, sticky table column headers, sticky sidebars.
--z-dropdown20 Float z-dropdown Dropdown menus, select popovers, date pickers, command palette.
--z-popover25 Float z-popover Tooltips, hover cards. Sits above dropdowns so tooltips show over open menus.
--z-overlay30 Overlay z-overlay Modal backdrop / scrim. Does not cover toasts.
--z-modal40 Modal z-modal Dialog, AlertDialog, Sheet (drawer), full-screen overlays.
--z-toast50 Toast z-toast Sonner toasts, snackbars, inline notifications. Always on top of modals.
--z-skip-nav99 A11y z-skip-nav Skip-to-content link. Accessibility requirement — must be topmost.
Why z-25 for popovers? Tooltips must appear above open dropdown menus. A user hovers over a truncated option inside a Select — the tooltip needs to sit above the dropdown (z-20) but below the overlay (z-30). The +5 gap solves this without restructuring the scale.

05 · Tailwind Config

Complete Configuration

Copy-paste into globals.css and tailwind.config.ts. This replaces any existing z-index configuration.

/* ── globals.css — Add to :root ─────────────────────────── */ :root { --z-base: 0; --z-raised: 1; --z-sticky: 10; --z-dropdown: 20; --z-popover: 25; --z-overlay: 30; --z-modal: 40; --z-toast: 50; --z-skip-nav: 99; }
/* ── tailwind.config.ts — theme.extend.zIndex ──────────── */ zIndex: { 'base': 'var(--z-base)', 'raised': 'var(--z-raised)', 'sticky': 'var(--z-sticky)', 'dropdown': 'var(--z-dropdown)', 'popover': 'var(--z-popover)', 'overlay': 'var(--z-overlay)', 'modal': 'var(--z-modal)', 'toast': 'var(--z-toast)', 'skip-nav': 'var(--z-skip-nav)', }
/* ── Usage in components ────────────────────────────────── */ /* MainHeader */ className="sticky top-0 z-sticky" /* Select dropdown content */ className="z-dropdown" /* Dialog overlay */ className="fixed inset-0 bg-black/50 z-overlay" /* Dialog content */ className="fixed z-modal" /* Sonner Toaster */ className="z-toast"
06 · Scenarios

Common Stacking Scenarios

Real situations where stacking order matters in VendHub, and how the token system resolves each one.

Dropdown opens from sticky header
The header is z-sticky (10). A navigation dropdown from the header uses z-dropdown (20). The dropdown naturally appears above the header content. No z-fighting.
Tooltip inside a dropdown
User hovers a truncated option inside a Select. The tooltip uses z-popover (25), which sits above z-dropdown (20). The tooltip is visible above the open menu.
Modal with toast notification
A modal opens (z-modal 40) with its scrim (z-overlay 30). A toast fires (z-toast 50). The toast appears above the modal — the user can see the notification while the modal is open.
Dropdown inside a modal
A Select inside a Dialog opens its dropdown. Since both the modal (z-modal 40) and the dropdown (z-dropdown 20) are inside the same stacking context created by the modal, the dropdown naturally appears on top without needing a higher z-index. This is CSS stacking context behavior — children inherit their parent's layer.
Card hover overlapping adjacent card
When a card's hover shadow grows (elevation L2 → card-hover), it might clip under the adjacent card. Adding z-raised (1) on hover ensures the hovered card's shadow renders above its neighbors.
Skip-to-content link
The skip navigation link is visually hidden but becomes visible on keyboard focus. It uses z-skip-nav (99) to guarantee it appears above everything — including toasts and modals. This is a WCAG requirement.
07 · Rules

Do's & Don'ts

✓ Do
Always use named z-index tokens: z-sticky, z-dropdown, z-modal, z-toast. The name communicates intent and makes refactoring safe.
✗ Don't
Never use raw numbers: z-[999], z-[9999], z-50 (Tailwind's default). These are magic numbers that create escalation wars. If one developer writes z-[999], the next writes z-[9999].
✓ Do
Let CSS stacking contexts do the work. A dropdown inside a modal doesn't need a higher z-index than the modal — it inherits the modal's stacking context.
✗ Don't
Don't set z-index on elements that don't need it. Most content sits at z-base (0) by default. Adding z-index to regular content creates unnecessary stacking contexts that can cause bugs.
✓ Do
Keep the overlay (scrim) and modal content on separate layers: z-overlay (30) for the backdrop, z-modal (40) for the content. This ensures the scrim darkens everything below while the modal sits above it.
✗ Don't
Don't put the toast container inside a modal's DOM tree. Toasts should be rendered at the root level (<Toaster /> in layout.tsx) so their z-toast isn't trapped inside a modal's stacking context.
✓ Do
Use isolation: isolate (Tailwind: isolate) to create new stacking contexts intentionally when you need to scope z-index behavior to a component.
✗ Don't
Don't use z-auto or negative z-indices (z-[-1]). Negative z-values create confusing stacking behavior and make debugging harder.
08 · Governance

Z-Index Governance

Rules for maintaining the z-index system as new components are added.

When to add a new token
Only add a new z-index token if a component genuinely cannot use one of the 9 existing tokens. The gap between tiers (e.g. z-25 between z-20 and z-30) exists specifically for this purpose. Adding a token requires documenting the use case and updating this file.

Example: If a floating toolbar needs to sit between sticky (10) and dropdown (20), use --z-toolbar: 15 and add it to the scale.
PR checklist
1. No raw z-index numbers — run grep -r "z-\[" --include="*.tsx" to catch bracket values.
2. No Tailwind default z-index classes (z-10, z-20, z-50) — use named tokens instead (z-sticky, z-dropdown, z-toast).
3. Toast container is at root level, not inside modal DOM.
4. Skip-nav link exists and uses z-skip-nav.
Cross-reference: This z-index system works with the Shadow/Elevation system (L1–L5) — higher z-index layers typically use higher elevation shadows. It also works with the Opacity system — the modal overlay uses bg-black/50 (opacity-50) at z-overlay.

Why this matters for VendHub

Without named z-index tokens, developers guess values and create stacking bugs. Here are 4 real scenarios where the system prevents problems.

Scenario 1
Dropdown from sticky header
A user clicks "Orders" in the header. The dropdown (z-dropdown: 20) must appear above the sticky header (z-sticky: 10) that contains it. Without defined layers, the dropdown can render behind the header.
Scenario 2
Tooltip inside a dropdown
A user hovers over a truncated option in a Select. The tooltip (z-popover: 25) must appear above the dropdown (z-dropdown: 20) so it's readable. The +5 gap between tiers handles this.
Scenario 3
Toast notification during modal
A user confirms an order (modal open at z-modal: 40) and a previous order's confirmation toast fires (z-toast: 50). The toast must be visible above the modal — critical for order confirmations.
Scenario 4
The z-index war (without system)
Developer A writes z-[999] for a dropdown. Developer B writes z-[9999] for a modal. Developer C writes z-[99999] for a toast. Named tokens eliminate this escalation entirely.
